= Composition
:page-title: Composition
:page-description: Let agent.yml decide which capabilities are built into your agent, by generating the pom before the build.

Qlawkus lets a single manifest, `agent.yml`, decide which capabilities are composed into your agent. The manifest is the source of truth for the build: its `build-time` section generates the application pom's dependency block, so a capability you do not ask for is never even on the classpath.

[NOTE]
====
This feature is landing in stages. The manifest schema, the generator (`mvn qlawkus:generate`), the capability catalog, and the `runtime` toggle tier are in place: the generator reconciles the pom against the capabilities the reactor's extensions declare, and the `runtime` block is applied at boot with an external override. Still being wired: dev-mode hot reload (issue #77).
====

== Why a generator

Quarkus resolves Maven dependencies *before* augmentation runs, and Maven does no tree-shaking, so a build step cannot add or drop dependencies for its own build. The only place a manifest can decide the dependencies is a step that runs *ahead* of the build. Qlawkus does this with a Maven plugin, the same idea as `quarkus extension add`:

[source,text]
----
agent.yml  ->  mvn qlawkus:generate  ->  pom.xml  ->  mvn package (or quarkus build --native)
----

A deselected capability is left out of the pom entirely, so the result is lean by construction - not by relying on native-image dead-code elimination.

== The manifest

[source,yaml]
----
version: 1

build-time:
  # Posture of every capability:
  #   enabled  -> all on,  list the ones to turn OFF
  #   disabled -> all off, list the ones to turn ON
  default: enabled
  # Capabilities with the opposite effect of the default. One list.
  except:
    - brag
    - skill-hub

runtime:
  qlawkus.skill-hub.approval-mode: hitl
  qlawkus.agent.semantic-extractor.enabled: true
----

`version`:: Schema version. Only `1` is accepted; it lets the schema evolve without breaking older files.

`build-time.default`:: `enabled` or `disabled`. The posture for all capabilities. Pick the default that makes the `except` list shortest.

`build-time.except`:: The capabilities that get the *opposite* effect of `default`. A single list whose meaning inverts with the posture (policy + exceptions, like a firewall default-DENY/ALLOW or `.gitignore` + `!`).

`runtime`:: Toggles applied at boot (see <<Runtime toggles>>). Each key is a full Quarkus configuration property name, used verbatim - `qlawkus.skill-hub.approval-mode`, `qlawkus.agent.semantic-extractor.enabled` - so there is no prefix translation to guess. Only capabilities already compiled in can be toggled here; adding or removing a capability is a build-time concern.

By default the manifest lives at `src/main/resources/qlawkus/agent.yml`. Point elsewhere with the build-time property `qlawkus.composition.manifest`.

== Running the generator

[source,bash]
----
mvn qlawkus:generate
----

The plugin reads the manifest, resolves the selected capabilities, and reconciles the pom's `<dependencies>` block in place: it adds the selected capabilities, removes the deselected ones, and leaves everything else (parent, plugins, container-image - the skeleton) untouched. It is idempotent, so a pom already in sync is left byte-identical. Capability dependencies are added with no version, managed by the platform BOM in your pom skeleton, exactly as `quarkus extension add` does.

In a container, run the generator in the build stage before `mvn package` or `quarkus build`. Swapping capabilities in a running production instance is a rebuild-and-redeploy, not a live toggle (the same closed-world constraint as a native image); the `runtime` section is the part that changes without a rebuild.

== Runtime toggles

The `runtime` block is the tier that changes without recompiling: it toggles among capabilities *already compiled in*, re-read each boot. It never adds or removes a capability - that is the build-time tier above - so it is the mirror of the secrets layering, with the opposite intent: secrets outrank the environment, runtime toggles yield to it.

Three layers contribute a value, lowest to highest ordinal:

[cols="1,1,3"]
|===
|Layer |Ordinal |Where

|Baked-in defaults |250 |The `runtime:` block of the bundled `agent.yml`, shipped in the artifact.
|External override |290 |A standalone YAML map of toggles on the filesystem, outside the artifact.
|Environment / `-D` |300+ |Ordinary Quarkus configuration.
|===

So a persisted override wins over a baked default, but a deploy-time environment variable or `-D` system property still wins over both - the operator always has the last word. The external override is a plain map of the same verbatim keys, kept on a mounted volume so it survives a rebuild of the image:

[source,yaml]
----
# ~/.qlawkus/agent.runtime.yml
qlawkus.skill-hub.approval-mode: yolo
----

Live changes persist here, never back into the baked artifact. At boot the agent logs which overrides are in effect (or that none are), and a missing or malformed override is reported, not fatal.

Knobs (`qlawkus.runtime.*`): `enabled` (default `true`), `override-path` (default `~/.qlawkus/agent.runtime.yml`), `manifest` (classpath location of the baked manifest, default `qlawkus/agent.yml`), and `baked-ordinal` / `override-ordinal` to retune the layering. Lower `override-ordinal` below 300 is already the default; raise it above 300 only if you want a persisted toggle to outrank the environment.

== Capabilities and where they come from

The catalog of capability names is not a central list. Each extension module *self-describes*: it declares its capability name in its own `quarkus-extension.yaml` under `metadata.qlawkus.capability`. The generator reads that key from every module in the build, so adding a new capability is just a matter of shipping a module that announces one - no central vocabulary to edit.

[source,yaml]
----
# in an extension's src/main/resources/META-INF/quarkus-extension.yaml
metadata:
  qlawkus:
    capability: "brag"
----

Names are namespaced where natural, and several modules may share one name (selecting or deselecting it moves them as a unit). The capabilities that ship today:

[cols="1,3"]
|===
|Capability |Provided by

|`cognition.pgvector` |The Postgres/pgvector backend for the cognition stores
|`messaging.discord` |Discord adapter
|`messaging.telegram` |Telegram adapter
|`messaging.slack` |Slack adapter
|`messaging.whatsapp` |WhatsApp adapter
|`brag` |Career brag-document tool
|`skill-hub` |Remote skill-registry client
|`google-workspace` |The six Google modules (auth, Gmail, Calendar, Drive, Sheets, Storage)
|===

The core agent (`qlawkus-client`) is the skeleton, always present, so it declares no capability. A capability name in the manifest's `except` list that no module announces is surfaced as a warning (fail-fast policy is decided in issue #77).
