= Composition
:page-title: Composition
:page-description: Let agent.yml decide which capabilities are built into your agent, by generating the pom before the build.

Qlawkus lets a single manifest, `agent.yml`, decide which capabilities are composed into your agent. The manifest is the source of truth for the build: its `build-time` section generates the application pom's dependency block, so a capability you do not ask for is never even on the classpath.

[NOTE]
====
The manifest schema, the generator (`mvn qlawkus:generate`), the capability catalog, the `runtime` toggle tier, and dev-mode reload are all in place: the generator reconciles the pom against the capabilities the reactor's extensions declare, unknown names fail the build, the resolved set is reported, and the `runtime` block is applied at boot with an external override. Composing a capability in or out is a build-time concern by design (a rebuild, the same closed-world trade a native image makes); the `runtime` section is the part that changes without one.
====

== Why a generator

Quarkus resolves Maven dependencies *before* augmentation runs, and Maven does no tree-shaking, so a build step cannot add or drop dependencies for its own build. The only place a manifest can decide the dependencies is a step that runs *ahead* of the build. Qlawkus does this with a Maven plugin, the same idea as `quarkus extension add`:

[source,text]
----
agent.yml  ->  mvn qlawkus:generate  ->  pom.xml  ->  mvn package (or quarkus build --native)
----

A deselected capability is left out of the pom entirely, so the result is lean by construction - not by relying on native-image dead-code elimination.

== The manifest

[source,yaml]
----
version: 1

build-time:
  # Posture of every capability:
  #   enabled  -> all on,  list the ones to turn OFF
  #   disabled -> all off, list the ones to turn ON
  default: enabled
  # Capabilities with the opposite effect of the default. One list.
  except:
    - brag
    - skill-hub

runtime:
  qlawkus.skill-hub.approval-mode: hitl
  qlawkus.agent.semantic-extractor.enabled: true
----

`version`:: Schema version. Only `1` is accepted; it lets the schema evolve without breaking older files.

`build-time.default`:: `enabled` or `disabled`. The posture for all capabilities. Pick the default that makes the `except` list shortest.

`build-time.except`:: The capabilities that get the *opposite* effect of `default`. A single list whose meaning inverts with the posture (policy + exceptions, like a firewall default-DENY/ALLOW or `.gitignore` + `!`).

`runtime`:: Toggles applied at boot (see <<Runtime toggles>>). Each key is a full Quarkus configuration property name, used verbatim - `qlawkus.skill-hub.approval-mode`, `qlawkus.agent.semantic-extractor.enabled` - so there is no prefix translation to guess. Only capabilities already compiled in can be toggled here; adding or removing a capability is a build-time concern.

By default the manifest lives at `src/main/resources/qlawkus/agent.yml`. Point elsewhere with the build-time property `qlawkus.composition.manifest`.

== Running the generator

[source,bash]
----
mvn qlawkus:generate
----

The plugin reads the manifest, resolves the selected capabilities, and reconciles the pom's `<dependencies>` block in place: it adds the selected capabilities, removes the deselected ones, and leaves everything else (parent, plugins, container-image - the skeleton) untouched. It is idempotent, so a pom already in sync is left byte-identical. Capability dependencies are added with no version, managed by the platform BOM in your pom skeleton, exactly as `quarkus extension add` does.

Each run logs the resolved capability set (what it selected, what it left out), so the manifest's policy-and-exceptions shorthand never hides the full picture.

If the manifest's `except` list names a capability that no module declares, the build fails: a typo would otherwise drop the extension silently, since a name nothing announces just resolves to nothing. Set `-Dqlawkus.composition.fail-on-unknown-capability=false` to downgrade that to a warning, for a partial reactor build where not every module is present.

In a container, run the generator in the build stage before `mvn package` or `quarkus build`. Swapping capabilities in a running production instance is a rebuild-and-redeploy, not a live toggle (the same closed-world constraint as a native image); the `runtime` section is the part that changes without a rebuild.

Either way you need the toolchain - Maven and a JDK. The manifest decides *what* is built; it never removes the *build*, because Quarkus is closed-world. That toolchain does not have to be on your host, though: the multi-stage build images (`app/src/main/docker/Dockerfile.jvm-build` and `Dockerfile.native-build`, driven by `./run.sh`) carry Maven and the JDK in the build stage and run the generator there. So `./run.sh prod build` (or `docker compose build`) rebuilds from the manifest with only Docker installed - no host Maven. Use the host `mvn` path when you already have the toolchain; use the Docker path when you do not.

== Dev-mode reload

Bind the goal to your app build and Quarkus dev mode keeps the pom in sync with the manifest as you edit it:

[source,xml]
----
<plugin>
  <groupId>dev.omatheusmesmo</groupId>
  <artifactId>qlawkus-maven-plugin</artifactId>
  <version>...</version>
  <executions>
    <execution>
      <id>compose</id>
      <goals>
        <goal>generate</goal>
      </goals>
    </execution>
  </executions>
</plugin>
----

The goal's default phase is `generate-sources`. Because `agent.yml` lives under `src/main/resources`, a directory Quarkus dev mode already watches, editing it triggers a live reload; the bound goal regenerates the pom, and Quarkus, which also watches `pom.xml`, restarts with the new capability set. It reuses two watchers that already exist rather than adding one. This is heavier than a code reload (a full dev restart), but it is automatic.

[NOTE]
====
Capability resolution reads the reactor (`${reactorProjects}`), so run dev mode from the reactor root (for example `mvn -pl app -am quarkus:dev`), not a single-module `cd app && mvn quarkus:dev`: a lone module sees an empty catalog and has nothing to compose. The `runtime` toggles below hot-reload either way, since they are configuration, not dependencies.
====

== Runtime toggles

The `runtime` block is the tier that changes without recompiling: it toggles among capabilities *already compiled in*, re-read each boot. It never adds or removes a capability - that is the build-time tier above - so it is the mirror of the secrets layering, with the opposite intent: secrets outrank the environment, runtime toggles yield to it.

Three layers contribute a value, lowest to highest ordinal:

[cols="1,1,3"]
|===
|Layer |Ordinal |Where

|Baked-in defaults |250 |The `runtime:` block of the bundled `agent.yml`, shipped in the artifact.
|External override |290 |A standalone YAML map of toggles on the filesystem, outside the artifact.
|Environment / `-D` |300+ |Ordinary Quarkus configuration.
|===

So a persisted override wins over a baked default, but a deploy-time environment variable or `-D` system property still wins over both - the operator always has the last word. The external override is a plain map of the same verbatim keys, kept on a mounted volume so it survives a rebuild of the image:

[source,yaml]
----
# ~/.qlawkus/agent.runtime.yml
qlawkus.skill-hub.approval-mode: yolo
----

Live changes persist here, never back into the baked artifact. At boot the agent logs which overrides are in effect (or that none are), and a missing or malformed override is reported, not fatal.

Knobs (`qlawkus.runtime.*`): `enabled` (default `true`), `override-path` (default `~/.qlawkus/agent.runtime.yml`), `manifest` (classpath location of the baked manifest, default `qlawkus/agent.yml`), and `baked-ordinal` / `override-ordinal` to retune the layering. Lower `override-ordinal` below 300 is already the default; raise it above 300 only if you want a persisted toggle to outrank the environment.

== Staging a rebuild over the API

The running app cannot rebuild itself: a native binary is frozen, a JVM app is closed-world augmented at build time, and the runtime image ships no toolchain. So the composition admin endpoint never builds. It only records *intent* - a new manifest staged for the next rebuild - which an external builder acts on. That keeps it safe by construction: authenticated, and unable to execute anything beyond writing a validated manifest.

[cols="2,3"]
|===
|Endpoint |What it does

|`POST /api/admin/composition/manifest` |Validate an `agent.yml` (same parser as the generator) and, only if valid, stage it. A malformed manifest is rejected with `400` and never staged.
|`GET /api/admin/composition` |The `active` manifest (baked into the running app) and the `staged` one pending a rebuild.
|`DELETE /api/admin/composition/manifest` |Discard the staged manifest.
|===

All three require authentication. The staged manifest lives in the app's own state (`qlawkus.agent.state.root`), not a shared volume: the seam to whatever rebuilds and rolls out the new image is this API, read over HTTP, rather than a filesystem both containers mount.

== The restart contract

Staging a manifest records intent; it never rebuilds. Closing the loop - turning a staged manifest into a running instance with the new capability set - is a rebuild and a restart driven from *outside* the app, because the app cannot rebuild or restart itself (a native binary is frozen, a JVM app is closed-world augmented at build time, and the runtime image ships no toolchain).

Qlawkus defines that as a five-phase contract and ships reference implementations under `examples/redeploy/`. It is never a hard dependency: the consuming environment wires the actual mechanism.

[cols="1,4"]
|===
|Phase |What happens

|1. fetch |Read the staged manifest over `GET /api/admin/composition`.
|2. promote |Write it into the source tree the builder reads (`app/src/main/resources/qlawkus/agent.yml`).
|3. build |Regenerate the pom from the manifest (`qlawkus:generate`) and build the artifact.
|4. restart |Swap the running instance onto the new artifact.
|5. verify |Wait for health, then discard the consumed staged manifest (`DELETE`).
|===

The app stays passive: it only fetches (`GET`) and discards (`DELETE`); everything that executes a build or a restart lives in the redeploy tooling, never in the app. Phases 1, 2 and 5 are the environment-independent hook (`examples/redeploy/redeploy.sh`); phases 3 and 4 are the parts each reference implementation fills in:

[cols="1,4"]
|===
|Environment |Build + restart step

|docker-compose |`docker compose up -d --build --force-recreate --no-deps qlawkus` - build and recreate in one step (mirrors `run.sh restart`, standalone).
|Kubernetes |Build and push an image, then `kubectl set image` + `kubectl rollout status` (the readiness probe gates the swap).
|systemd |Package on the host, then `systemctl restart qlawkus`.
|===

Because the staged manifest lives inside the app's own state and is read back over the API, the redeploy tooling - not the app - is what promotes it into the source before a build. If nothing is staged, a redeploy simply rebuilds the currently committed manifest, so it is safe to run in CI on every deploy.

== Capabilities and where they come from

The catalog of capability names is not a central list. Each extension module *self-describes*: it declares its capability name in its own `quarkus-extension.yaml` under `metadata.qlawkus.capability`. The generator reads that key from every module in the build, so adding a new capability is just a matter of shipping a module that announces one - no central vocabulary to edit.

[source,yaml]
----
# in an extension's src/main/resources/META-INF/quarkus-extension.yaml
metadata:
  qlawkus:
    capability: "brag"
----

Names are namespaced where natural, and several modules may share one name (selecting or deselecting it moves them as a unit). The capabilities that ship today:

[cols="1,3"]
|===
|Capability |Provided by

|`cognition.pgvector` |The Postgres/pgvector backend for the cognition stores
|`messaging.discord` |Discord adapter
|`messaging.telegram` |Telegram adapter
|`messaging.slack` |Slack adapter
|`messaging.whatsapp` |WhatsApp adapter
|`brag` |Career brag-document tool
|`skill-hub` |Remote skill-registry client
|`google-workspace` |The six Google modules (auth, Gmail, Calendar, Drive, Sheets, Storage)
|===

The core agent (`qlawkus-client`) is the skeleton, always present, so it declares no capability. A capability name in the manifest's `except` list that no module announces fails the build by default (see <<Running the generator>>).
