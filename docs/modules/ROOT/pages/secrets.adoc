= Secrets
:page-title: Secrets
:page-description: Keep secrets in an encrypted keystore on a volume instead of plain environment variables, with no database.

Qlawkus can read secrets from an encrypted PKCS12 keystore on a mounted volume instead of plain environment variables, with no datasource involved. The `qlawkus-client` extension wires this up by convention, so it works in a database-free distribution and inside a container.

== How it works

Each entry (alias) in the keystore is named after the configuration property it supplies. A stored alias transparently overrides the matching property, so consumers keep reading an ordinary `@ConfigProperty` and need no change. For example, an alias `quarkus.langchain4j.openai."primary".api-key` supplies the LLM key that would otherwise come from `LLM_API_KEY`.

Resolution is lenient when the file is absent: with no keystore present the agent boots normally on your `.env`, and the feature stays inert until you drop a keystore in place.

== Precedence

`qlawkus-client` publishes keystore secrets at config ordinal `qlawkus.secrets.keystore-ordinal` (default `350`), so a stored secret is authoritative:

[source,text]
----
system properties (400)  >  KEYSTORE (350)  >  env vars (300)  >  .env (295)  >  application.properties (250)
----

A secret in the keystore therefore wins over the matching environment variable, so the rule is simply: drop it in the keystore and it takes effect, with no stray env default able to shadow it. Only a system property (`-D...`) still overrides, which keeps an emergency operator override available. If you would rather let an environment variable override a stored secret, lower `qlawkus.secrets.keystore-ordinal` below `300`.

== Onboarding a secret

Add a secret with `keytool`, naming the alias after the property it overrides:

[source,bash]
----
keytool -importpass \
  -alias 'quarkus.langchain4j.openai."primary".api-key' \
  -keystore ~/.qlawkus/secrets.p12 \
  -storetype PKCS12
----

`keytool` prompts for the secret value and for the keystore password. Repeat once per secret. List what is stored with:

[source,bash]
----
keytool -list -keystore ~/.qlawkus/secrets.p12 -storetype PKCS12
----

== What to store

The alias is the configuration property the consumer reads, so move a secret out of your `.env` by storing it under the matching property name below. Each entry is otherwise sourced from the environment variable in the last column.

[cols="2,3,2",options="header"]
|===
|Secret |Keystore alias (config property) |Env equivalent

|LLM API key |`quarkus.langchain4j.openai."primary".api-key` |`LLM_API_KEY`
|Embedding API key |`qlawkus.embedding.api-key` |`LLM_EMBEDDING_API_KEY`
|API basic-auth password |`quarkus.security.users.embedded.users.qlawkus` |`API_USER_PASSWORD`
|Google OAuth client secret |`qlawkus.google.auth.client-secret` |`GOOGLE_CLIENT_SECRET`
|Google token-vault passphrase |`qlawkus.google.vault.encryption-passphrase` |`QLAWKUS_VAULT_PASSPHRASE`
|Discord bot token |`qlawkus.messaging.discord.bot-token` |`DISCORD_BOT_TOKEN`
|Telegram bot token |`qlawkus.messaging.telegram.bot-token` |`TELEGRAM_BOT_TOKEN`
|Transcription (Whisper) key |`qlawkus.messaging.transcription.api-key` |`WHISPER_API_KEY`
|TTS provider key |`qlawkus.messaging.tts.providers.<id>.api-key` |`TTS_<ID>_API_KEY`
|Datasource passwords |`quarkus.datasource.password`, `quarkus.datasource."google-auth".password`, `quarkus.datasource."brag".password` |`QUARKUS_DATASOURCE_*_PASSWORD`
|===

The consumer needs no change in every case: the keystore (ordinal 350) outranks the `application.properties` value (250) that otherwise reads the environment variable. The LLM and embedding keys are routed through an internal kind factory that runs below the keystore and reads the stored value back, so they resolve from the keystore exactly like a direct property. The friendly `LLM_API_KEY` / `LLM_EMBEDDING_API_KEY` names also work as aliases if you prefer them over the native property names.

== Unlocking the keystore

Provide the keystore password through the environment and never commit it:

[source,bash]
----
export QLAWKUS_SECRETS_KEYSTORE_PASSWORD=...   # the storepass you set above
----

== Path and containers

The keystore lives at `qlawkus.secrets.keystore-path`, default `~/.qlawkus/secrets.p12`. In a container, mount a host directory to that path on a persistent volume so secrets survive restarts, and pass `QLAWKUS_SECRETS_KEYSTORE_PASSWORD` as an environment variable. The keystore file and its password stay out of the image.

== Managed alternative: Vault

For a centrally managed store, put `quarkus-vault` on the classpath. It contributes a `CredentialsProvider` and resolves secrets the same way (as ordinary config), so consumers are unchanged. Use the keystore for a self-contained, offline deployment and Vault when secrets are managed centrally.
